本章面向需要阅读源码、扩展功能、构建镜像或参与项目维护的开发人员。

开发前先区分两类工作：一种是产品功能开发，另一种是部署、品牌、文档和示例资源维护。前者需要同时考虑后端接口、前端页面、权限、审计、数据迁移和测试；后者也要走构建验证，不建议只改页面或配置后直接发布。

开发调试时建议从真实页面入口理解功能。工作台是 Crest 的用户入口，数据准备、仪表盘、数据大屏、系统管理等功能都从这里进入。

![工作台](/images/docs/crest/screenshots/workspace.png)

新增功能时，要考虑它最终会出现在什么菜单、需要什么权限、是否会进入审计日志、是否需要补充用户文档和截图。

## 源码结构

| 路径 | 说明 |
| --- | --- |
| `core/core-backend` | Spring Boot 后端服务，包含接口、业务实现、Flyway 迁移和最终 JAR 打包入口 |
| `core/core-frontend` | Vue 3.3、Vite、TypeScript、Element Plus、Pinia 前端工程 |
| `sdk/api` | 对内 API、DTO、VO 和接口契约 |
| `sdk/common` | 认证、通用模型、工具类、异常处理和 Spring 配置 |
| `sdk/extensions/extensions-datasource` | JDBC 数据源扩展和方言能力 |
| `drivers` | 随仓库维护的 JDBC 驱动 |
| `installer` | 单机安装脚本、Docker Compose 模板、控制脚本和离线包制作脚本 |
| `deploy/kubernetes` | Kubernetes 部署清单 |
| `docs` | 项目说明、开发说明和专题文档 |

阅读源码时建议从业务入口反向追踪：用户在页面上做什么，前端调用哪个接口，接口落到哪个服务，服务是否访问数据库、缓存、文件或外部系统。这样比直接逐个浏览目录更容易理解调用链路。

## 后端本地启动

先准备 MySQL，并设置环境变量：

```bash
export CREST_DB_HOST=127.0.0.1
export CREST_DB_PORT=3306
export CREST_DB_NAME=crest
export CREST_DB_USERNAME=root
export CREST_DB_PASSWORD='<mysql-password>'
export CREST_AES_KEY='<32-character-aes-key>'
export CREST_AES_IV='<16-character-aes-iv>'
export CREST_CRYPTO_MODE='standard'
export CREST_SM4_KEY='<16-character-sm4-key-when-crypto-mode-is-sm-suite>'
export CREST_INITIAL_PASSWORD='<admin-initial-password>'
./run.sh start
```

后端启动时会通过 Flyway 执行数据库迁移。迁移脚本应保持环境无关，不要写入个人账号、内网地址、真实密码或临时资源。

本地开发建议使用独立数据库，不要连接共享测试库或生产库。调试数据源、数据集和权限时，可以造小规模示例数据，确保字段和业务口径可复现。

## 本地国密调试

需要验证 v1.5.0 国密能力时，将本地环境切换为：

```bash
export CREST_CRYPTO_MODE='sm-suite'
export CREST_SM4_KEY='0123456789abcdef'
```

`CREST_SM4_KEY` 必须是 16 位 Key。切换后建议重新启动后端，并至少验证登录、修改密码、创建数据源、保存数据集、打开仪表盘、打开数据大屏、分享访问和导出下载。国密模式不改变前端操作流程，但会影响登录传输、密码哈希、敏感配置密文和下载票据签名等底层链路。

## 前端本地启动

```bash
cd core/core-frontend
pnpm install --frozen-lockfile
pnpm dev
```

前端页面主要包括工作台、数据源、数据集、数据血缘、仪表盘、数据大屏、分享、导出中心、修改密码和系统管理。

前端调试时建议准备两个账号：管理员和普通用户。仅使用管理员账号测试，容易遗漏菜单权限、资源权限、只读状态和普通用户入口。

## 构建命令

前端构建：

```bash
cd core/core-frontend
pnpm run build:base
```

前端轻量检查：

```bash
cd core/core-frontend
pnpm run build:lite:check
```

后端打包：

```bash
mvn -s .mvn/settings.xml -pl :core-backend -am clean package -Pstandalone -DskipTests -Dmaven.test.skip=true
```

本地镜像：

```bash
docker build -t ghcr.io/sevoniva/crest:local .
```

## 菜单与页面

Crest 的主菜单由后端 `core_menu` 表驱动，前端根据后端返回的菜单生成路由。新增公开菜单时，通常需要：

1. 新增前端页面组件。
2. 在数据库迁移中新增菜单记录。
3. 确认菜单权限默认值。
4. 更新角色权限和资源权限逻辑。
5. 补充用户文档和截图。

不应只在前端写死菜单，否则权限、路由、刷新和后续升级会出现不一致。

## 接口开发

新增接口时建议遵循：

- 请求参数使用明确的 DTO。
- 返回给前端的错误信息可理解，不泄露内部堆栈、密码和完整连接串。
- 涉及用户、角色、资源和数据源的接口必须检查权限。
- 导出、下载、分享和 SSO 等外部入口要记录审计日志。

## 数据库迁移

迁移脚本位于后端资源目录下的 Flyway 目录。原则：

1. 脚本按版本递增。
2. 初始化数据不能包含生产敏感信息。
3. 变更字段类型前评估历史数据。
4. 删除表、字段或菜单前提供兼容方案。
5. 演示数据和产品边界要同步更新文档。

## 扩展开发建议

常见扩展方向：

| 方向 | 建议 |
| --- | --- |
| 新增数据源类型 | 优先复用现有 JDBC 数据源扩展机制 |
| 新增图表 | 先确认图表配置、渲染、导出和大屏兼容 |
| 新增系统菜单 | 同步处理菜单权限、角色默认值和文档 |
| 接入统一认证 | 优先扩展现有 SSO 配置 |
| 调整品牌 | 修改站点设置、Logo、登录页和文档资源 |

扩展功能时要同步考虑文档和截图。Crest 是面向用户使用的数据产品，新增入口、按钮、配置项和权限后，如果缺少文档说明，用户难以稳定复现操作，也会增加后续支持成本。

## 提交前检查

提交代码前至少确认：

1. 没有提交 `.env`、数据库密码、Token、证书或生产导出文件。
2. 前端构建或轻量检查通过。
3. 后端涉及模块可编译。
4. 数据库迁移能在空库执行。
5. 新增功能有对应文档或说明。
6. 公开仓库中没有不可分发依赖和授权不清的资源。
